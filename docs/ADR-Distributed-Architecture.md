# Distributed Architecture Decision Record

## Context
The Real-Time Financial Monitor needs to scale horizontally across multiple pods/replicas. When deployed to Kubernetes with multiple replicas, clients connected to different pods won't receive real-time updates from other pods.

## Problem
```
Client A ??? Pod 1 ??? SQLite (local)
Client B ??? Pod 2 ??? SQLite (local)
Client C ??? Pod 3 ??? SQLite (local)
```

When a transaction is sent to **Pod 1**, only clients connected to **Pod 1** via SignalR will receive the update. Clients on **Pod 2** and **Pod 3** will miss the update.

## Solution: SignalR Backplane

### Option 1: Azure SignalR Service (Recommended for Production)

**Pros:**
- ? Fully managed - no infrastructure to maintain
- ? Auto-scaling based on connection count
- ? Built-in geo-replication
- ? 99.9% SLA
- ? Supports millions of concurrent connections

**Implementation:**
```csharp
// Program.cs
builder.Services.AddSignalR()
    .AddAzureSignalR(builder.Configuration["Azure:SignalR:ConnectionString"]);
```

### Option 2: Redis Backplane (Self-Hosted)

**Pros:**
- ? Open source - no vendor lock-in
- ? Lower cost (self-hosted)
- ? Full control over infrastructure
- ? Can use existing Redis cluster

**Cons:**
- ?? Requires Redis maintenance
- ?? Need to handle Redis HA/clustering yourself

**Implementation:**
```csharp
// Program.cs
builder.Services.AddSignalR()
    .AddStackExchangeRedis(builder.Configuration["RectionString"]);
```

**NuGet Package:**
```bash
dotnet add package Microsoft.AspNetCore.SignalR.StackExchangeRedis
```

### Option 3: Message Queue (Event-Driven)

For even better scalability, use **Azure Service Bus** or **RabbitMQ**:

```csharp
// Publish transaction events
await serviceBus.PublishAsync(new TransactionCreatedEvent(dto));

// All pods subscribe and broadcast to their connected clients
```

## Database Strategy for Multi-Pod

### Current: SQLite (Single-Pod Only)
- ? Simple for MVP
- ? File-based - can't share across pods
- ? Not suitable for distributed systems

### Production: Shared Database
```csharp
// PostgreSQL / SQL Server / Azure SQL
builder.Services.AddDbContext<AppDbContext>(opt =>
{
    opt.UseNpgsql(configuration["ConnectionStrings:PostgreSQL"]);
    // or
    opt.UseSqlServer(configuration["ConnectionStrings:SqlServer"]);
});
```

## Decision: Start with Redis, Migrate to Azure SignalR

1. **Phase 1 (MVP)**: SQLite + Local SignalR ? (Current)
2. **Phase 2 (Multi-Pod)**: PostgreSQL + Redis Backplane
3. **Phase 3 (Scale)**: Azure SQL + Azure SignalR Service
